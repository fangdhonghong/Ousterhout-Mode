# Ousterhout Mode

> **“投资 20% 的开发时间用于软件设计，以换取系统长期的可维护性。”**

`Ousterhout Mode` 是一个专门为深度代码重构设计的 AI Skill（针对 Claude Code 等工具优化）。它不追求“快速修复”，而是严格遵循 John Ousterhout 教授在《软件设计之道》（*A Philosophy of Software Design*）中提出的原则，旨在通过**战略编程（Strategic Programming）**消除系统复杂性。

### 核心理念
市面上大多数 AI 工具都处于“战术编程”阶段——即为了完成当前任务而采取最小改动，这往往会导致复杂性的累积。本 Skill 强制将开发流程切换至“战略模式”，通过构建**深模块（Deep Modules）**和**隐藏细节**，从根本上降低系统的认知负荷。

---

### 核心功能

#### 1. 深度红旗扫描 (Contextual Audit)
AI 不再提供泛泛的修改建议，而是使用书中精确定义的术语进行“红旗（Red Flags）”定位。它会识别出隐藏的**信息泄漏**、**透传方法**以及由于**时间分解**导致的架构漏洞。

#### 2. 设计两次 (Design It Twice)
在修改任何核心逻辑前，该 Skill 会强制要求 AI 提供两个截然不同的设计方案（方案 A 与 方案 B）。这种做法能打破 AI 的思维惯性，通过对比不同抽象层级的优劣，选出认知负担最低的最优解。

#### 3. 注释先行 (Documentation as Design)
回归“注释即设计”的本质。在编写重构代码前，AI 必须先写出重构后的接口注释。如果无法用简洁的语言描述清楚，说明该抽象依然是“浅”的，必须重新设计。

#### 4. 消除异常 (Define Errors Out of Existence)
不只是捕获异常，而是尝试重新定义接口语义，让边缘情况变为正常流程，从而降低调用者的心智负担。

---

### 运行流程

1. **状态对齐**：自动运行 `git status` 确保工作区整洁，并强制建议开启重构分支。
2. **红旗清单**：列出当前代码中最严重的架构缺陷，并解释其为何违反设计原则。
3. **方案博弈**：针对核心问题展示两种重构方案，进行深度对比。
4. **增量交付**：以小步迭代的方式执行修改，每一步重构均需开发者通过 Diff 确认。
5. **投资报告**：重构完成后，总结本次优化的“设计资产”与未来建议。

---

### 红旗参考手册

| 术语 | 表现 | 治理目标 |
| :--- | :--- | :--- |
| **浅模块 (Shallow Module)** | 接口复杂，实现简单 | 增加模块深度，隐藏复杂性 |
| **信息泄漏 (Information Leakage)** | 实现细节散落在多个地方 | 封装知识，降低全局耦合 |
| **透传方法 (Pass-Through)** | 方法只是简单转发请求 | 合并模块，减少无效抽象 |
| **类炎 (Classitis)** | 为了拆分而拆分，产生大量无意义的小类 | 适当合并，构建更深的模块 |
| **时间分解 (Temporal Decomposition)** | 按照执行顺序划分代码 | 按照知识和意图重新组织结构 |

### 如何启动

如果你使用的是支持该 Skill 定义的环境，只需输入：

```bash
启动重构 [模块路径]
```

或者直接说：

```bash
用 Ousterhout Mode 审查一下这段逻辑，重点看接口深度。
```

---

### 为什么要做这个？
软件设计的挑战不在于如何写出能运行的代码，而在于如何管理随规模增长而爆炸的复杂性。`Ousterhout Mode` 将 AI 从一个“打字员”变成了一个“架构陪练”，它不仅帮你改代码，更在过程中强迫你思考：**这个抽象，真的值得吗？**
